# Release process

How releases on `sc-agent-public` happen, with review.

> **Status v0**: single maintainer is the reviewer of record. The flow below is shaped so a multi-reviewer model can drop in later (community moderators, vote-based review, ML pre-screening — whatever the project ends up wanting) without changing day-to-day mechanics.

## Three release tracks

The repo publishes three distinct kinds of releases. Each has its own naming, its own contents, and its own cadence — but they share the same review flow.

| Track | Tag pattern | Cadence | Triggered by |
|---|---|---|---|
| **App** | `app-vX.Y.Z` | Per app version | Maintainer when an MSIX is ready |
| **Dataset** | `dataset-v<sc-patch>` | Per Star Citizen patch | Maintainer when CIG ships a patch and the build pipeline catches up |
| **Models** | `models-vX.Y.Z` | Rare (when training improves) | Maintainer when a new model bundle is benched and signed |
| **Phase** | (committed to `main` directly, then optionally tagged `phase-vYYYY-MM-DD`) | Periodic aggregation | Maintainer aggregates accumulated community contributions |

## Flow per release (any track)

### 1. Open a PR

Even the maintainer opens a PR — direct push to `main` is reserved for trivial fixes (typos, doc links). Releases always go through a PR so:
- The CI verification workflow runs on the proposed change.
- The diff is visible for sanity check (especially for dataset releases — sometimes a coverage regression sneaks in).
- A reviewer (currently the maintainer; future: whatever shape the reviewer pool takes) can comment on the proposal.

Branch naming convention:
```
release/dataset-v4.7.3
release/models-v0.0.2
release/app-v0.0.1
```

PR title mirrors the branch.

### 2. CI verifies

Workflows that run automatically on PRs to `release/*` branches:

- **`verify-manifest.yml`** — for model releases, validates the manifest's Ed25519 signature against the embedded trust anchor. **Blocks merge** if the signature doesn't verify.
- **`update-stats.yml`** (dry-run) — for dataset / phase changes, regenerates the README stats block and shows the diff in the PR. Doesn't block merge; just gives the reviewer a preview.
- **Schema lint** (TBD — when there's enough schema to be worth it) — validates `phase/*.json` and `universe/*.json` against their JSON Schemas.

If a check fails, the PR description should explain why before requesting review (sometimes a fail is intentional — e.g., schema bump).

### 3. Review

The reviewer (currently the maintainer; future: see the [Decisions log → reviewer model](dev/decisions-log.md) once that decision is made) walks through:
- **Diff sanity** — does the change actually look like what the PR says it does?
- **Numbers** — for dataset releases, do the stats look right? (Containers count, POI count, system count — sudden drops usually mean a pipeline regression.)
- **Signatures** — for model releases, is the manifest signed by the expected key version?
- **Privacy** — for phase data merges, scan for accidentally-included PII (the app should anonymize, but human eyes catch things automated checks miss).
- **Consumer URL stability** — does this break any documented `releases/download/...` URL? If so, is it called out in CHANGELOG.md and the release notes?

Review happens via GitHub PR comments. If a comment surfaces a real issue, the contributor (or maintainer) pushes a fix to the PR branch; CI re-runs; cycle until green.

### 4. Merge + tag

Once approved + CI green:

1. Squash-merge the PR to `main` with a clean commit message: `Release dataset-v4.7.3` (or `Release models-v0.0.2`, etc.).
2. Tag the merge commit with the release tag: `git tag dataset-v4.7.3 && git push --tags`.
3. **Update rolling tag** if applicable (`dataset-latest`, `models-latest`).
4. **Create the GitHub Release** from the tag, attach the artifact files (dataset.json, model bundles, etc.), paste release notes from `release-notes.md` (auto-generated by the build pipeline).

Steps 1–4 are scriptable; see `tools/dataset-builder/publish-dataset.ps1` and the unified runbook in the private code repo (`docs/operations/public-repo-release.md`).

### 5. Announce

For each release:
- **App** → pinned Discussion in the **Announcements** category (manual write-up — what's new, known issues, upgrade path).
- **Dataset** → Discussion comment on the prior dataset announcement, or a new Announcement if the patch brings something significant (new system, big POI delta).
- **Models** → Announcement only if the user-visible behavior changes (e.g., faster, more accurate). Patch releases can skip.
- **Phase** → no announcement per release; periodically post a "phase data update" Announcement summarizing several aggregations at once.

Tone of announcements: same as the README — fan project, engineering taken seriously, plain English. Don't write release notes in the third person ("The maintainer has..." sounds weird for a single-dev project).

## Reviewer model — where this is going

Today: maintainer is the only reviewer. The CODEOWNERS file lists `@zimm1` as owner of every path.

Later (when it makes sense): community reviewers for specific data areas — e.g., a Pyro expert reviews `universe/pyro/*` PRs, a phase-math expert reviews `phase/*` aggregations. CODEOWNERS supports this with per-path entries.

Even later, if the project's vision pushes that way: ML-assisted pre-review (auto-flag suspicious diffs), vote-based merge thresholds (community Discussion vote required for big architectural changes), or whatever shape the project decides. **The current flow is shaped so any of those drop in without changing the day-to-day mechanics.**

When a reviewer-model decision lands, it goes in [`dev/decisions-log.md`](dev/decisions-log.md) with rationale.

## What does NOT go through this flow

- Typo fixes in README / docs — direct push to `main` is fine.
- Discussion replies, issue comments, label changes — none of these are PRs.
- Workflow runs themselves (the bot commits with `[skip ci]` to avoid review loops on auto-generated content).

## Hotfix path

If something is on fire (e.g., a published manifest fails verification on user machines):

1. Open the PR with `release/hotfix-...` branch.
2. Add `urgent` label.
3. Reviewer fast-tracks within hours, not days.
4. Same merge + tag flow; release notes call out the hotfix.

There's no separate "emergency push to main" path — hotfix still goes through review. The cost of one extra approval round is much less than the cost of a hotfix that itself breaks something.
